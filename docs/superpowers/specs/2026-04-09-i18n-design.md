# Xiaoyun i18n Design — Chinese/English Bilingual Support

## Goal

Add Chinese/English language switching to Xiaoyun so the open-source release is accessible to English-speaking users. Language auto-detects from the OS and can be overridden in Settings.

## Scope

- **In scope**: All frontend UI text (labels, buttons, placeholders, error messages, status text, tooltips, relative time expressions)
- **Out of scope**: AI prompts in Rust backend (`prompts.rs`, `wiki_prompts.rs`) — these stay in Chinese. AI model output language is unaffected.

## Architecture

### Library

**react-i18next** + **i18next** + **i18next-browser-languagedetector**

- `i18next`: core i18n engine
- `react-i18next`: React bindings (useTranslation hook, Trans component)
- `i18next-browser-languagedetector`: auto-detect OS/browser language on first launch

### i18n Initialization

New file: `src/lib/i18n.ts`

```typescript
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import LanguageDetector from 'i18next-browser-languagedetector';
import { listen } from '@tauri-apps/api/event';

// Import all translation files statically (small app, no lazy loading needed)
// ... (all 12 imports)

i18n
  .use(LanguageDetector)
  .use(initReactI18next)
  .init({
    supportedLngs: ['zh-CN', 'en'],
    nonExplicitSupportedLngs: true, // zh-TW → zh-CN, en-US → en
    resources: { /* ... */ },
    fallbackLng: 'en',              // unknown locales → English (open source default)
    defaultNS: 'common',
    interpolation: { escapeValue: false },
    detection: {
      order: ['localStorage', 'navigator'],
      lookupLocalStorage: 'openwiki-language',
      caches: ['localStorage'],
    },
  });

// Keep <html lang> in sync
i18n.on('languageChanged', (lng) => {
  document.documentElement.lang = lng;
});

// Cross-window sync: listen for language change events from other windows
listen<{ language: string }>('language-changed', (event) => {
  const lng = event.payload.language;
  if (lng === 'auto') {
    i18n.changeLanguage(undefined);
  } else {
    i18n.changeLanguage(lng);
  }
});

export default i18n;
```

Import in `src/main.tsx` before app render:

```typescript
import './lib/i18n'; // must be before App import
```

### Language Persistence

**Storage**: SQLite via settingsStore (same as theme, capture mode, etc.)

- `settingsStore.ts` gets a new `language` field (`'zh-CN' | 'en' | 'auto'`), persisted via `updateSetting('language', lng)`
- On first launch: `i18next-browser-languagedetector` reads OS language → `supportedLngs` + `nonExplicitSupportedLngs` maps `zh*` to `zh-CN`, everything else to `en`
- When user changes language in Settings: `setLanguage()` saves to SQLite, calls `i18n.changeLanguage()`, and broadcasts a Tauri event `language-changed` to all windows
- localStorage is used as a fast cache by i18next-browser-languagedetector, but SQLite is the source of truth — `loadFromDB` syncs localStorage on startup

### Settings UI — Language Switcher

Add a "语言 / Language" option as the **first item in the Appearance (外观) category**, since language and theme are both appearance-level preferences.

Design:

```
外观
  语言 / Language     [  中文  |  English  ]    ← segmented control, same style as theme picker
  主题模式            [  浅色  |  深色  | 跟随系统  ]
```

The language label is always shown bilingually ("语言 / Language") so English users can find it even if the UI is currently in Chinese.

Implementation:

```tsx
const { setLanguage, language } = useSettingsStore();

// Language switcher — goes through settingsStore which persists to SQLite + broadcasts Tauri event
<SegmentedControl
  value={language === 'auto' ? (i18n.language.startsWith('zh') ? 'zh-CN' : 'en') : language}
  onChange={(lng) => setLanguage(lng)}
  options={[
    { value: 'zh-CN', label: '中文' },
    { value: 'en', label: 'English' },
  ]}
/>
```

### Component Migration Pattern

Each component replaces hardcoded Chinese text with `t()` calls:

**Before:**
```tsx
const TABS = [
  { id: "content", label: "内容", icon: ClipboardList },
  { id: "wiki", label: "知识", icon: BookOpen },
];
```

**After:**
```tsx
const { t } = useTranslation();

const TABS = [
  { id: "content", label: t('nav.content'), icon: ClipboardList },
  { id: "wiki", label: t('nav.wiki'), icon: BookOpen },
];
```

For components with namespace-specific text:

```tsx
const { t } = useTranslation('settings');
// t('appearance.theme') → "主题模式" or "Theme"
```

### Data-Driven Labels (settingsStore)

Store data objects (`PROVIDER_LABELS`, `MODELS_BY_PROVIDER`) keep their current shape unchanged. The UI layer derives translation keys from stable IDs using convention:

```tsx
// In SettingsView — derive key from provider ID
const label = t(`provider.${provider}`, { defaultValue: PROVIDER_LABELS[provider] });

// In model selector — derive key from group string
const groupLabel = t(`model.group.${normalizeKey(model.group)}`, { defaultValue: model.group });
```

Translation files provide the mappings:
```json
// zh-CN/settings.json
{ "provider.dashscope": "阿里云百炼", "model.group.free.recommended": "免费推荐" }

// en/settings.json
{ "provider.dashscope": "Alibaba DashScope", "model.group.free.recommended": "Free (Recommended)" }
```

The store never stores i18n keys as data values.

### Brand Name

Brand name is "OpenWiki" in both languages. The header shows "OpenWiki" regardless of language setting. This replaces the previous "小云" branding as part of the open-source release.

### html lang Attribute

Update `<html lang>` reactively when language changes:

```typescript
// in i18n.ts, after init
i18n.on('languageChanged', (lng) => {
  document.documentElement.lang = lng;
});
```

### Tauri System Tray Menu

The tray menu is defined in Rust (`lib.rs` or similar). Tray menu items like "显示主窗口", "退出" need bilingual support. Approach:

- On app startup and language change, the frontend sends the current language to the backend via a Tauri command
- The Rust side stores the language preference and rebuilds the tray menu with the correct language
- This is a **separate, optional phase** — tray menu can stay Chinese initially and be translated in a follow-up

### Floating Bubble (BubbleView)

The bubble window (`BubbleView.tsx`) is a separate Tauri WebView window. It imports from the same `src/` codebase, so it shares the same i18n setup. Each window's `i18n.ts` registers a Tauri event listener for `language-changed`, so when the main window calls `setLanguage()`, all other windows receive the event and call `i18n.changeLanguage()` to update live. On window creation, the i18n instance reads the initial language from the detector (localStorage cache synced from SQLite on startup).

## Migration Strategy

Work module-by-module. Each module ships with BOTH zh-CN and en translations together. Language switcher lands LAST (after all modules are migrated) to avoid exposing a mixed-language app during migration.

1. **Foundation** — Install deps, create `i18n.ts`, create all 12 translation JSON files (both languages, complete), key parity script, SQLite language field, import in `main.tsx`
2. **App shell** — Migrate `App.tsx` (nav tabs, search, brand rename to OpenWiki)
3. **Content module** — Migrate content list, cards, bubble, spotlight
4. **Wiki module** — Migrate all wiki components
5. **Digest module** — Migrate insight/radar components
6. **DataHub + Report** — Migrate data hub and weekly report components
7. **Settings** — Add language switcher + migrate all settings text (switcher enabled LAST)
8. **Polish** — Key parity check, search for remaining hardcoded text, smoke test both languages

## Edge Cases

- **English text is often longer than Chinese** (e.g., "设置" → "Settings", "内容捕获" → "Content Capture"). UI must accommodate with flexible layouts — avoid fixed-width containers for text. Current Tailwind layout uses `flex` and `truncate` which handles this well.
- **Number formatting**: Chinese "3天" vs English "3 days" — handled by i18next interpolation `{{count}}`.
- **Segmented controls**: Buttons with text like "浅色|深色|跟随系统" may need slightly wider min-width for English "Light|Dark|System". Minor CSS adjustment.
- **Missing translation fallback**: `fallbackLng: 'en'` ensures unknown locales default to English. Since both zh-CN and en are always complete (enforced by parity check), missing keys should not occur in practice.

## Testing

- **Key parity check**: `npx tsx scripts/check-i18n-keys.ts` must exit 0 — all zh-CN and en files have identical key sets
- **Hardcoded text scan**: grep for Chinese characters in .tsx/.ts files (excluding locales/, comments, console.log) — should return empty
- Switch language in Settings → all visible text updates immediately, no reload needed
- Close and reopen app → language preference persists (SQLite-backed)
- First launch on English OS → UI shows English
- First launch on Chinese OS → UI shows Chinese
- Switch language in main window → bubble/spotlight windows update live (Tauri event sync)
- Backend error surfaces show translated generic message, not raw Chinese error string
- No console warnings about missing translation keys
- Layout does not overflow or break in either language (check: header nav, settings sidebar, segmented controls, bubble)
